Design Decisions
→
"Add Client" is the primary CTA, top-right of the table — consistent with Arch's existing pattern (e.g., "Invite Accountant" in Tax Docs).
→
Stat cards give the advisor a snapshot of their book before diving in — mirrors the Holdings and Tax Docs summary cards.
→
The table follows exact Arch conventions: checkbox column, sortable headers (↕), blue hyperlinked names, status tags, action column.
2
Step 1 of 3 — Create Client
Clicking "Add Client" opens a focused modal. Step 1 is deliberately minimal — just the client name — so advisors can move quickly. A stepper at the top orients them within the 3-step flow.

Design Decisions
→
Modal keeps the user in context (they can still see the Clients table behind the overlay) rather than navigating to a new page — reduces perceived complexity for a 2-field step.
→
Stepper communicates the 3-step structure upfront so the advisor knows exactly what's ahead.
→
The optional "Internal Reference" field anticipates advisor workflows — advisors often tag new clients with source notes.
3
Step 2 of 3 — Create Entities
After naming the client, the advisor adds their legal entities inline — one row per entity. All 5 can be filled in at once without leaving the page. The "+ Add Entity" pattern scales from 1 to hundreds without changing the UX.

Design Decisions
→
Inline table editing avoids modals-within-modals. Advisors can see all entities at once and edit without leaving context — critical for catching naming inconsistencies across a family's entities.
→
"+ Add Entity" at the bottom grows the list naturally. Rows are numbered so the advisor can track their count.
→
CSV import is surfaced but not forced — power users can bypass row-by-row input at scale, while new advisors get the guided path.
→
Moving to a full page (vs. modal) for Step 2 gives the table room to breathe — especially important when an advisor has 20+ entities to add.
4
Step 3 of 3 — Assign User Permissions
A user × entity permission matrix lets the advisor assign access at a glance. "Select all" toggles handle the common case (full access), while individual checkboxes handle exceptions like Pierre and Eve's scoped access.

Design Decisions
→
Matrix layout (users as rows, entities as columns) makes the full permission state visible at once — no need to open each user individually.
→
"Select All" per column handles the common case (Advisor, Accounting, Marie Curie) in a single click — reduces friction for the majority pattern.
→
Individual checkboxes handle exceptions (Pierre: 3 of 5, Eve: 1 of 5) without any additional UI complexity.
→
Entity type is shown under each column header — helps advisors quickly verify they're assigning access to the right entity type.
5
Review & Confirm
Before submitting, the advisor sees a clean summary of everything they've configured — client name, all entities, and the permission assignments. Each section has an "Edit" link to jump back if anything needs correcting.

Design Decisions
→
The confirmation screen is the advisor's "last look" before data is written — reducing support tickets from accidental submissions. Each section has an inline Edit link rather than requiring a full back-navigation.
→
Permissions are summarized intelligently: "All Entities" for the simple case, "Custom (N of 5: …)" for scoped access — readable without showing the full matrix again.
→
The primary CTA changes to green to signal a commitment action — distinct from the blue "Continue" buttons used throughout the flow.
→
Entity type tags (Individual, Trust, LLC) use distinct colors — helpful for advisors scanning across many entities of different types.
