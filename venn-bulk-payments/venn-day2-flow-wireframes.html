01.2

Screen Inventory

S0 — Entry / Initiate

The trigger point. Access from Send button or sidebar nav. Sets context before the user touches a file.

S1 — Pre-Upload

Template download, column spec, sample CSV. Sets user up for success before they upload.

S2 — Upload

Drag-and-drop zone. Tier 1 format validation runs on drop. Immediate feedback on file type / parse issues.

S3 — Parsing & Validating

Loading state with progress copy. Tier 2 business-logic scan runs. Manages wait-time anxiety.

S4 — Review Table

The centrepiece. All rows, per-row status, inline error display, error filter, summary bar, toolbar. Inline editing.

S4-B — Method Selection

Triggered when a recipient has 2+ payment methods. Inline drawer or mini-modal to pick one per row.

S4-E — Inline Repair

Error rows expanded, editable. Error count badge updates live. "Send clean rows" option if some errors remain.

S5 — Confirmation

Final summary: total outflow, per-currency breakdown, fees, funding account selector, FX rate lock. Big confirm CTA.

S5-F — Insufficient Funds

Surfaced at confirm. Shows shortfall, options: top up, remove rows, or proceed with affordable subset.

S6 — Sending…

Processing state. Progress indicator. Don't abandon the user mid-flight.

S7 — All Sent

Full success. Per-payment confirmation. Download receipt. Link to transaction history.

S7-E — Partial Failure

Mixed outcome. Clear split: sent / processing / failed. Failed rows have a retry path or download.

Four screens that get the user from "I need to pay multiple people" to "my file is parsed and ready to review." The key design principle here: set expectations before the user touches a file. Most products skip the pre-upload step and pay for it in failed uploads and re-tries.

Entry point from the Send button context menu or sidebar

1

Error types are distinct: Red = hard error (missing required field), Amber = soft warning (needs user decision). Different colors, different severity, different actions.

2

"Errors only" filter is active by default when errors exist. User can toggle to "All (12)" if they want context. This pattern reduces cognitive load — don't force scanning 12 rows to find 3 problems.

5

"Confirm all" is visually disabled while errors remain — grayed out, not-allowed cursor. But it's still visible so users understand what they're working toward.

S7-E — Partial Failure
Mixed outcome — some sent, some failed. The most important post-send state to design well.
Don't hide failure. Show the exact split. Give failed rows a retry path. Let the user download the failure report so they can fix and re-upload. This screen is often the difference between a user trusting your product and filing a support ticket.

Annotations
1

Outcome grid first — user immediately sees 9 / 1 / 2 split. Numbers are the headline; failure reasons are secondary. Don't bury the lede.

2

Failure reason is specific per row: "Account number rejected by receiving bank" vs. "Recipient bank closed for holiday" — these require different user actions. One needs a fix, one needs a retry tomorrow.

3

"Funds returned to your account" confirmation in the red banner — the user's most immediate anxiety after a failure is "is my money gone?" Answer it immediately.

4

Download failure report (.csv) — lets users fix their original data and re-upload as a new batch. Closes the loop back to S1.

5

"Retry failed payments" bulk action — queues the 2 failed rows as a new micro-batch. No manual re-entry required.
