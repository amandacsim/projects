01.1

The Universal Bulk Import Framework

🔑 Industry-standard pattern (from UX research)

Every well-designed bulk workflow follows the same 5 stages: Pre-import → Upload → Mapping → Repair → Import. This is the mental model your design should follow — even if you compress some steps for Venn's simpler use case. The biggest failure mode is skipping the Repair stage or making users start over when errors are found.

01 · Pre-import

Set Guardrails

Provide a downloadable template and clear format guidance before upload. Reduces errors at source. Show column requirements, accepted currencies, example rows.

02 · Upload

File Ingestion

Drag-and-drop + browse + copy-paste. Immediate parsing feedback. Show row count and detected columns. Flag format issues instantly — don't wait until the next step.

03 · Mapping

Column Review

Auto-map CSV headers to expected fields. Allow manual correction. Show a preview of parsed data so users can catch issues before validation runs.

04 · Repair ← Most Critical

Error Resolution

Flag errors inline per row. Allow editing without leaving the flow. Show "errors only" filter. Many products fail here by dumping a generic error message.

05 · Confirm

Final Summary + Send

Show total amount, recipient count, currency breakdown, funding account. One clear CTA. This is a financial commitment — make the summary feel like a receipt.

01.3

Key UX Patterns to Adopt & Avoid

✓ Adopt — High Signal
  • Downloadable CSV template with example row before upload starts
  • Inline error display per row — users should never have to re-upload to fix one field
  • "Show errors only" filter toggle on the review table
  • Summary bar: total amount, # recipients, # errors, currencies involved
  • FX rate lock with countdown timer (Wise's rate-urgency pattern)
  • Readiness check gate — can't confirm until all fixable errors are resolved
  • Irreversibility warning at confirmation (appropriate financial gravity)
  • Partial success state — show what sent, what failed, what to do next
✗ Avoid — Low Signal or Poor UX
  • Generic error: "Your file is invalid" — always be specific about what and where
  • Forcing users to re-upload the entire CSV to fix one row
  • Async batch-job model with no real-time feedback (Currencycloud pattern)
  • Blocking the entire batch if one row has an error — give users control
  • Showing all 500 rows at once — use pagination + smart defaults (errors first)
  • Hiding the total cost/fees until the confirmation screen
  • Multi-modal error reporting (some inline, some toast, some modal) — pick one

⚡ The Venn-Specific Challenge

Most competitors (Wise, Rippling, Tipalti) deal with homogeneous batches — same currency, same payment type, same recipient structure. Venn's brief explicitly says rows may have different currencies AND different payment methods. This is harder to design for. The table needs to surface method and currency per row without overwhelming the user. This is where your design can genuinely stand out.

02.1

Problem Statement & Goal

Problem Statement

Business owners at Venn currently have no way to pay multiple recipients at once. Sending 20 payments means completing the single-payment flow 20 times — a process that compounds cognitive load, error risk, and time cost with every additional recipient. The CSV upload workaround exists, but as a raw technical tool with no validation, no visibility, and no recovery path when things go wrong.

Design Goal

Design an end-to-end bulk payments experience that feels as clear and trustworthy as sending a single payment — while scaling to handle dozens or hundreds of recipients across multiple currencies and payment methods. The experience must make complexity feel manageable, not overwhelming.

02.2

The User

Primary User

The Business Owner / Founder

Typically wears multiple hats. Does their own finances and payments because they don't have a dedicated ops team yet. Time-sensitive. Comfortable with spreadsheets. Wants control, but doesn't want to babysit each payment.

Secondary User

The Finance / Ops Manager

At slightly larger SMBs. Owns the payments process. More systematic. Cares about audit trails and reconciliation. Will likely be the one who prepares the CSV in advance and executes the batch.

Context (Brief)

Already a Venn user

Familiar with Venn's single-payment flow. Has mental models around how transfers are labeled, how currencies work on the platform. Doesn't need onboarding — needs an extension of what they already know.

02.4

User Priorities & Pain Points

Must Feel → Trustworthy
  • Financial transactions carry anxiety — users need reassurance at each step
  • Showing totals, fees, and recipient details before confirmation is non-negotiable
  • Any ambiguity about "what am I about to send?" kills trust
  • Irreversibility needs to be acknowledged, not hidden
Must Feel → In Control
  • Users want to be the decision-maker, not a data clerk
  • Errors should never block the entire batch — give users choices
  • Option to remove a row vs. fix it, or send the clean rows and queue the rest
  • Clear confirmation of what's been selected for sending
Pain Point → Error Recovery
  • The current CSV upload has no recovery — it's a black box
  • Users are used to fixing things in spreadsheets — inline editing feels natural
  • Biggest fear: sending wrong amount to wrong recipient with no way to undo
  • Second biggest fear: spending 30 mins on a batch only to have it rejected
Pain Point → Complexity at Scale
  • Mixed currencies and payment methods per row are hard to mentally track
  • FX rates for different currencies change — need to know the rate locked in
  • Insufficient funds for one currency vs. another complicates the flow
  • Recipients with multiple payment methods adds selection complexity
Nice to Have → Speed
  • Faster than doing it one by one (table scan is faster than 20 individual forms)
  • "Send clean rows now, fix errors later" option for time-sensitive batches
  • Batch label / memo for internal tracking
  • Re-use last batch template as starting point
Tension to Design For
  • Speed vs. Safety — faster flow vs. more confirmation steps
  • Simplicity vs. Completeness — hide complexity vs. show all data
  • Control vs. Guidance — let users decide vs. system makes smart defaults
  • Error strictness — block on any error vs. allow partial sends
02.5

User Journey

Trigger
User decides to pay multiple recipients
It's payroll day, or end of month, or a batch of vendor invoices. User opens Venn and wants to pay 15–100 people at once. They have the data in a spreadsheet already.
Mental model: "I need to do 50 payments but I can't bear to do them one at a time. I hope Venn has a better way."
Entry
Navigate to Bulk Payments
User finds the bulk payments entry point — either from the main nav, from the "Send" button context, or from a nudge on the single-payment flow if they're on it.
Pain: "Is it just CSV upload? Do I need to format my spreadsheet a certain way? What columns does it expect?"
Prep
Download template / format their CSV
User downloads the Venn CSV template or checks column requirements. Maps their existing data to the required format. Saves as CSV.
Pain: "My spreadsheet has different column names. Did I get the format right? I won't know until I upload and it either works or breaks."
Upload
Upload the CSV file
Drag-and-drop or browse to upload. System parses the file, shows row count, detects columns. Immediate feedback that the file was parsed successfully.
Pain: "Did it work? How many rows did it find? Are the columns right? I'm nervous until I see something on screen."
Review
Scan and validate the payment table
System displays all rows in a table with validation status. Shows summary stats at top: X payments, Y currencies, $Z total, N errors. User scans for issues.
Pain: "There are 3 errors — what's wrong with them? Can I fix them here or do I need to go back to my spreadsheet? What about the clean rows?"
Repair
Fix errors in the table
User clicks into error rows, edits fields inline, or removes rows they can't fix now. Error count drops. User may choose to proceed with clean rows only.
Pain: "The error message says 'invalid account details' but doesn't tell me what's wrong. Row 14 has a USD amount but a CAD account — do I need to pick a different source account?"
Confirm
Review summary and confirm the batch
User sees a final confirmation screen: total outflow, breakdown by currency, funding account balance check, fee total. Selects funding account if multiple accounts. Hits confirm.
Pain: "My USD balance is $4,200 but the USD total is $5,000. What do I do? Can I send the ones I can afford and defer the rest?"
Post-Send
Batch in progress / success / partial failure
System processes the batch. User gets a clear outcome: X payments sent, Y processing, Z failed (with reasons). Failed payments have a retry or resolve path.
Pain: "3 payments failed — how do I retry them? Do I have to start a whole new batch? How do I know which ones are stuck vs. on their way?"
02.6

Solution Requirements & Constraints

Must Have
  • CSV upload with immediate parse feedback (row count, column detection)
  • Table view of all payment rows with per-row status (valid / error / warning)
  • Inline error display — specific message per row, per field if possible
  • Inline editing of rows without re-uploading
  • Summary bar: total, currencies, error count
  • Insufficient funds detection before confirmation
  • Funding account selection at confirmation
  • Clear confirmation screen before execution
  • Post-send state showing outcome per payment
Should Have
  • Multi-currency visibility per row (flag + currency code)
  • Multi-payment-method per row (badge for EFT, Wire, etc.)
  • "Show errors only" filter on the table
  • Recipient with multiple payment methods → method selection step
  • Option to send clean rows now, defer error rows
  • FX rate lock notice for multi-currency batches
  • Batch label / reference for internal tracking
  • Partial success screen with per-row outcome
Constraints
  • Must feel consistent with existing Venn single-payment flow (same terminology, same nav patterns)
  • Business owner persona — not a developer, not an accountant. Jargon-light copy.
  • Mobile is likely secondary — bulk payments are a desktop-first workflow
  • Trust is paramount — every design decision that saves a click at the cost of clarity is wrong
  • CSV is the stated input format — no Excel, no manual entry for the MVP
Trade-offs to Justify
  • Block vs. allow with errors: Allow sending clean rows while errors remain unresolved (more control) vs. block until all fixed (safer). Decision: allow, with explicit choice + warning.
  • Inline edit vs. modal edit: Inline (faster, more spreadsheet-like) vs. modal (more space for complex fields). Decision: inline for simple fields, modal/drawer for complex ones.
  • When to validate: During upload (realtime) vs. after upload (batch scan). Decision: hybrid — critical format errors during upload, business logic after. See Section 03.
Day 1 · Design Decisions

Key
Choices

Three fundamental design decisions to resolve before wireframing. Each one is shown as a comparison so you can pick confidently — and defend your choice in the final presentation.
03.1

Decision: Table View ✓ (Already Decided)

Why table is the right call — and how to defend it

Finance users think in rows and columns. Their source data lives in spreadsheets. A table view mirrors their mental model, reduces translation effort, and gives the highest information density per pixel. The alternatives (card grid, accordion list) trade density for visual spacing — wrong move when someone is scanning 50 rows for errors.

Table View ← Your Choice
  • Mirrors spreadsheet mental model of target users
  • Highest information density — scan 30 rows at a glance
  • Sortable, filterable by column
  • Easy to add status column, flag problematic rows
  • Familiar pattern for finance professionals
Card Grid (alternative)
  • Better for small batches (5–10 payments)
  • Each card can show full recipient detail richly
  • Poor at density — 50 cards is overwhelming
  • Hard to scan for errors across the batch
  • Wrong for this use case
Accordion List (alternative)
  • Hides detail by default, expand on click
  • Works for 10–20 items max
  • Poor for batch validation — errors hidden until expanded
  • Slow for scanning — requires interaction to see data
  • Wrong for this use case
Table Design Principles for this flow
  • Sticky header row as user scrolls
  • Row-level status indicator (colored left border or icon)
  • Summary bar pinned above the table (total, errors, currencies)
  • Errors-only filter toggle → reduces cognitive load
  • Inline edit on click — don't force a modal for simple fields
  • Currency flag + code inline in Amount column
  • Payment method badge per row (EFT, Wire, etc.)
  • Row checkbox for bulk actions (remove selected, etc.)
03.2

Decision: Error Display — Inline vs. Sidebar

Option A: Inline Errors
Recommended

Error messages appear directly within the table row, below the field with the issue. Row is highlighted (red left border, subtle background tint). User edits inline and error clears on resolution.

Errors are spatially located where the problem is — no context switching
Natural for users who think in spreadsheets (Excel shows errors in cells)
Editing and error reading happen in the same place
Scales well — 3 errors or 30 errors, same interaction model
Table rows with errors get taller — can disrupt scanning rhythm
Long error messages get truncated in narrow columns
Option B: Sidebar Errors
Alternative

A persistent right-panel shows a list of all errors. Each error item links to its row (clicking highlights or scrolls to it). Table rows are marked with an icon but the message lives in the sidebar.

Error list is scannable all at once — good for an error summary before fixing
Table rows stay consistent height — cleaner visual scanning
Works well when there are many errors (10+) to triage
Requires context switching — read error in sidebar, look for it in table
Sidebar takes horizontal space from the table (already space-constrained)
Disappears on mobile — not a progressively safe pattern
Two places to look — confusion about where to edit

Recommendation: Hybrid approach — two-tier validation

Tier 1 (immediate, during upload): File format checks only — wrong file type, unparseable CSV, missing required columns. These block the upload immediately with a clear error before the user even sees the table. Fast and synchronous.

Tier 2 (after upload, before table renders): Business logic validation — invalid account numbers, insufficient balance, currency mismatches, duplicate recipients, missing fields. This runs as a batch scan during the "Validating…" loading state, then the table renders fully validated.

Decision Choice Rationale (use this in your presentation)
Layout Table View Finance users think in spreadsheets. Highest information density. Scannable. Sortable/filterable. Right choice for any batch operation in a B2B fintech context.
